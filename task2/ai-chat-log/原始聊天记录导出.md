## 3. 采集策略 spec

### 采样窗口

默认每 60 秒生成一个 perf.data 文件：

```text id="x51c6q"
/data/perf/2026-06-11/20260611-030000.perf.data
/data/perf/2026-06-11/20260611-030100.perf.data
/data/perf/2026-06-11/20260611-030200.perf.data
```

文件名使用窗口开始时间，便于回查。

### perf record 命令

建议默认使用：

```bash id="hwieuu"
perf record -F 99 -a -g -- sleep 60
```

含义：

| 参数 | 说明 |
|---|---|
| `-F 99` | 99Hz 采样频率，避免过高开销，也避开 100Hz 周期性干扰 |
| `-a` | 全系统采样，适合捕捉未知故障进程 |
| `-g` | 采集调用栈，用于生成火焰图 |
| `sleep 60` | 每个窗口采集 60 秒 |

这是最适合作业场景的版本：简单、可解释、容易验证。

可配置环境变量：

```bash id="3d0osm"
WINDOW_SECONDS=60
RETENTION_HOURS=24
PERF_FREQ=99
DATA_DIR=/data/perf
OUTPUT_DIR=/data/output
```

## 5. 回查与火焰图生成 spec

### 输入

```bash id="o5xo3c"
perfbox flame \
  --from "2026-06-11 03:00:00" \
  --to   "2026-06-11 03:05:00" \
  --output /data/output/cpu-0300-0305.svg
```

### 行为

工具应做 4 步：

1. 解析用户输入时间段。
2. 找到所有与时间段有交集的 `.perf.data` 文件。
3. 对每个文件执行：

```bash id="okr9ti"
perf script -i xxx.perf.data
```

4. 拼接 `perf script` 输出，然后调用 FlameGraph：

```bash id="ko3oro"
stackcollapse-perf.pl > out.folded
flamegraph.pl out.folded > out.svg
```

### 产物

建议输出：

```text id="6l9h7d"
/data/output/
  cpu-20260611-030000-030500.svg
  cpu-20260611-030000-030500.folded
  cpu-20260611-030000-030500.perfscript
```

保留中间文件的好处是：如果 SVG 有问题，可以回看 folded 和 perf script，方便 debug。

## 8. README 必须写清楚的设计权衡

建议 README 单独写一节“设计权衡”：

### 为什么使用固定时间窗口，而不是一个巨大 perf.data？

因为持续运行一个大文件会导致：

1. 文件无限增长；
2. 故障时间点难以定位；
3. perf script 处理时间长；
4. 单文件损坏影响全部历史数据。

按 1 分钟窗口轮转，可以精确回查，也便于自动清理。

### 为什么默认全系统采样 `-a`？

因为故障发生前并不知道是哪一个进程导致 CPU 飙升。全系统采样更符合“黑匣子”定位目标。

### 为什么默认 99Hz？

99Hz 对大多数场景开销较低，同时足以定位 CPU 热点。太高频率会增加性能影响和磁盘压力。

### 为什么需要 `--privileged`？

因为 perf 需要访问宿主机内核性能事件、PMU 和进程调用栈信息。容器默认权限不足，所以必须使用特权模式。

### 为什么先做 CLI 而不是 API？

CLI 更简单、可复现、方便评审方验证，也完全满足题目“提供命令行接口或 API”的要求。
