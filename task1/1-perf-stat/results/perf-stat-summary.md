# perf stat 五场景横向对比

说明：本表基于锁核情况下的 `perf stat` 输出整理，取 `cpu_core/...` 行作为有效计数。由于当前 CPU 是 Intel 混合架构平台，`cpu_atom/...` 在本次锁核测试中显示为 `<not counted>` 或 `<not supported>`，未纳入计算。

## 2. 计算公式

| 衍生指标 | 计算方式 |
|---|---|
| IPC | `instructions / cycles` |
| L1 DCache Miss Rate | `L1-dcache-load-misses / cache-references` |
| LLC Miss Rate | `LLC-load-misses / cache-references` |
| 分支预测失败率 | `branch-misses / branch-instructions` |
| TLB Miss Rate | `dTLB-load-misses / cache-references` |

## 4. 备注

- `matrixprod` 的 L1 DCache Miss Rate 按题目公式计算后远高于 100%，原因是 `L1-dcache-load-misses` 与 `cache-references` 并不是严格同一语义层级的一组分子/分母；这里仍按题目要求的公式保留计算结果。
- `queens` 的分支预测失败率最高，为 11.595%，符合分支密集型负载的特征。
- `read64` 的 TLB Miss Rate 最高，为 9.698%，说明大块内存访问对地址翻译和页表/TLB 行为更敏感。

## 5. 微架构差异分析

本次测试平台为 Intel Core i9-13980HX，属于 Raptor Lake-HX 混合架构 CPU。由于需要进行横向对比，在混合架构下跑负载，任务可能在P核跑也可能在E核跑，对判断perf stat结果的变化是stress-ng负载本身的变化，还是跑在了不同的核心上。而且即使是在单一类型的核心上，Linux内核也可能在核心间迁移线程以平衡负载，这个迁移动作本身有成本，会导致CPU缓存（L1/L2）失效、TLB（转译后备缓冲器）被冲刷等，会对后面对比差异产生影响，所以选择锁核的操作。
锁核后本次 `perf stat` 的有效硬件事件主要来自 `cpu_core/...`，因此下面分析主要反映 P-core 上的执行特征。


### 5.1 纯计算整数负载：int64

`int64` 的 IPC 为 3.056，分支预测失败率只有 0.104%，LLC Miss Rate 为 1.985%，TLB Miss Rate 为 0.177%。这说明该负载主要由整数算术指令构成，控制流相对规律，前端取指、解码和分支预测压力都不高；指令能够较稳定地送入后端执行。

从后端看，整数 ALU 是主要执行资源。较高 IPC 表明流水线中有较多可并行发射和执行的整数指令，后端没有被长延迟内存访问严重阻塞。虽然按题目公式计算的 L1 DCache Miss Rate 为 70.512%，但原始 `cache-references` 规模很小，说明该负载整体并不是访存主导，cache miss 对总体执行时间的影响有限。综合判断，`int64` 更接近后端整数执行单元吞吐型负载，瓶颈主要在整数运算吞吐和指令依赖链，而不是内存层级。

### 5.2 浮点/矩阵负载：matrixprod

`matrixprod` 的 IPC 为 2.335，低于 `int64`，但分支预测失败率仍然很低，只有 0.152%。这说明它的控制流同样较稳定，前端分支预测不是主要瓶颈。矩阵乘法通常包含循环、乘加、数据加载和结果写回，热点更集中在浮点/SIMD 执行单元和数据供给之间的配合。

从后端执行单元看，浮点或向量乘加指令的延迟和吞吐约束比简单整数运算更明显。如果数据能稳定留在 cache 中，性能主要受浮点执行端口、向量化程度、循环展开和指令调度影响。本次 IPC 低于 `int64`，说明每条指令平均占用的执行资源更重，后端浮点/SIMD 单元或数据加载端口可能成为限制因素。

需要注意的是，`matrixprod` 的 L1 DCache Miss Rate 按题目公式达到 766599.334%，这是因为 `L1-dcache-load-misses` 与 `cache-references` 并不是严格匹配的一组分子和分母，不能简单理解为“每次 cache reference 都 miss 很多次”。但从原始计数看，`L1-dcache-load-misses` 达到 35,895,714,846，明显高于其他场景，说明矩阵计算中确实存在大量数据加载未命中 L1D 的情况。由于 LLC Miss Rate 只有 3.416%，说明很多未命中可能在 L2/L3 层级被承接，没有全部落到内存。综合判断，`matrixprod` 同时受到后端浮点/SIMD 执行吞吐和 L1/L2 数据供给能力影响。

### 5.3 连续访存负载：read64

`read64` 的 IPC 为 4.283，是五类负载中最高的，同时分支预测失败率仅 0.001%。这说明连续读取场景的控制流非常规则，前端能够稳定取指和解码，分支预测几乎没有造成流水线清空。高 IPC 并不一定表示单次内存访问延迟低，而是说明该负载可能具有较好的顺序访问特征，硬件预取器可以提前把数据搬入 cache 层级，CPU 能通过乱序执行和内存级并行隐藏一部分访存延迟。

访存子系统方面，`read64` 的 TLB Miss Rate 为 9.698%，显著高于其他负载，说明 1G 大块内存连续扫描带来了较高的地址翻译压力。连续访问对 cache 预取友好，但访问范围很大时会跨越大量页，dTLB 容量和页表遍历开销会变得明显。LLC Miss Rate 只有 0.558%，说明按 `cache-references` 口径看，最后一级缓存未命中比例并不高，顺序访问和预取机制有效降低了 LLC 层面的随机缺失。综合判断，`read64` 的主要压力不是前端或分支，而是大工作集下的地址翻译、内存带宽和预取效率。

### 5.4 随机访存负载：rand-set

`rand-set` 的 IPC 为 3.780，低于 `read64` 但高于 `int64` 和 `matrixprod`，分支预测失败率只有 0.008%。这说明它虽然是随机访存型负载，但控制流并不复杂，前端和分支预测并不是主要瓶颈。

从访存子系统看，`rand-set` 的 `cache-references` 达到 1,232,631,094，远高于其他负载，`cache-misses` 也达到 823,675,975，原始 cache miss 压力非常突出。随机写/随机设置类访问缺少稳定步长，硬件预取器难以预测后续地址，因此更容易破坏 cache 局部性，造成 cache 层级频繁替换。LLC Miss Rate 按题目公式为 1.119%，看起来不高，但结合大量 `cache-references` 和 `cache-misses` 原始计数，仍能说明该负载对 cache 子系统施加了最大规模的访问压力。与 `read64` 相比，`rand-set` 的 TLB Miss Rate 只有 0.030%，说明本次测试中瓶颈更偏向 cache 局部性和随机访问带来的 cache line 利用率下降，而不是地址翻译。

### 5.5 分支密集负载：queens

`queens` 的 IPC 为 1.953，是五类负载中最低的；分支预测失败率为 11.595%，明显高于其他负载。这是最典型的前端/控制流受限场景。N 皇后搜索包含大量递归、条件判断和回溯，不同搜索路径的数据相关性和控制流都更难预测。分支预测失败时，CPU 需要清空错误路径上的流水线，并从正确路径重新取指和解码，导致前端供给中断，后端执行单元也会出现空泡。

从后端看，`queens` 并不依赖大规模连续内存带宽，LLC Miss Rate 为 2.305%，TLB Miss Rate 为 0.165%，访存压力不是主要矛盾。它的核心瓶颈在分支预测和指令流控制：即使单条整数判断和位运算本身开销不大，频繁错误预测也会放大前端停顿成本，使 IPC 明显下降。因此该负载可以归类为前端分支预测/控制流瓶颈主导。

### 5.6 横向结论

从前端角度看，`queens` 最受分支预测影响，11.595% 的分支失败率直接解释了最低 IPC；`read64`、`rand-set`、`int64` 和 `matrixprod` 的分支失败率都很低，前端控制流不是主要瓶颈。

从后端执行角度看，`int64` 更偏整数 ALU 吞吐，IPC 较高；`matrixprod` 更依赖浮点/SIMD 执行单元和 load/store 端口，IPC 低于简单整数计算，说明单条指令占用资源更重，且数据供给对执行效率影响更明显。

从访存子系统角度看，`read64` 是连续大块访存，硬件预取较友好，但 TLB Miss Rate 最高，主要体现地址翻译和大工作集压力；`rand-set` 是随机访存，`cache-references` 和 `cache-misses` 原始计数最大，主要体现 cache 局部性差和预取失效；`matrixprod` 有大量 L1D miss，但很多访问可能仍在更低层 cache 中命中，属于计算与数据供给混合受限。

因此，五类负载的主导瓶颈可以概括为：`int64` 主要考察整数后端吞吐，`matrixprod` 主要考察浮点/SIMD 执行和 L1/L2 数据供给，`read64` 主要考察顺序访存、预取、TLB 和内存带宽，`rand-set` 主要考察 cache 局部性和随机访问代价，`queens` 主要考察前端分支预测和控制流恢复成本。
